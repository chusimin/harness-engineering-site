02 · WHY IT MATTERS

为什么

为什么 2026 起，harness 这件事决定 AI 产品胜负。

01 · NAKED vs HARNESSED

裸模型 vs 被驾驭的 Agent

同一个 Claude / GPT，套不套 harness 是两种产品。差距不在模型分数，差在"能干完一件事"。

NAKED MODEL · 裸模型
只能"想",不能"干"
×无状态:每次调用从零开始,记不住上一轮。
×不能调工具:只能输出文字,改不了文件、发不了请求。
×没有围栏:可能输出违规、泄密、删错文件,无人兜底。
×不可观测:出错了不知道哪一步坏的,没法定位。
×不能迭代:没评测集,改了 prompt 不知道是变好还是变差。
HARNESSED AGENT · 被驾驭的 Agent
能"干完",能"管住"
✓有记忆层:工作记忆 + 长期知识,跨轮、跨会话连续。
✓有工具集:文件、Shell、HTTP、子 agent,真能动手做事。
✓有沙箱与权限:危险动作必经审批,可回滚、可审计。
✓全链路 trace:每一步推理、工具调用、token 成本都被记下来。
✓有评测集 + A/B:每次改动都可量化,迭代有方向。

差距不在"模型多聪明",而在"被包裹了多少层工程"。 2026 年,各家旗舰模型在 benchmark 上已经接近,但Anthropic 的 Claude Code跑长任务的成功率仍能拉开 30%+ 的差距 —— 拉开它的不是模型,是 harness。

02 · INFLECTION POINT

行业拐点:2026 这一年

模型能力曲线在 2025 末刚刚跨过一道阈值,工程难度从 "调好 prompt" 跳到了 "撑起 8 小时长任务"。

~2024 · 短任务
≈ 40%
long-task → → 2026 · ≈ 71%
基于 METR 长时任务评测,旗舰模型连续完成 50+ 步工程任务的成功率,从 2024 到 2026 翻了近一倍。
~2024 · 工具调用准确度
≈ 65%
tool-use → → 2026 · ≈ 92%
OpenAI Codex 在 2026 报告中,工具调用一次命中率(无重试)突破 90%,工具编排正式可工程化。
~2024 · 单任务上下文
128k
context → → 2026 · 1M+
长上下文从 128k 跃升到 1M+,催生了 "整个 repo 喂进去" 的工作流;但上下文越长,harness 的成本和噪音控制越关键。

2026 年 2 月,OpenAI 在《Harness engineering》一文中第一次正式命名这件事;一个月后,Anthropic 发布《Harness design for long-running application development》,把 Claude Code 的 8 小时长任务工程公开。"harness" 一词,在两家头部实验室同一季度被独立提出 —— 这不是偶然命名,是模型能力溢出后,必须有的工程层。

过去三年,谁的 prompt 写得好,谁就赢;未来三年,谁的 harness 包得好,谁就赢。

03 · FIVE REASONS

5 个非做不可的理由

为什么 PM、工程师、组织,在 2026 起都必须把 harness 当一等公民。

01
模型能力溢出,需要 harness 装载

模型已经会"想",但用户要的是"做"。不套 harness,GPT-5 / Claude 4.6 也只是个聊天框 —— 装不进真实工作流。

参见 OpenAI 2026:"模型能力的瓶颈,正从能力本身转到 harness 上"。

02
长任务成功率由 harness 决定

50 步以上的任务,模型每步独立误差 5%,累计成功率不到 8%。靠 harness 的重试、check、回滚,才能拉到可商用水位。

Anthropic harness design 是这一点的实证案例。

03
成本治理需要 harness

一个失控的 agent,一晚上能烧掉一个团队一个月预算。token 配额、模型路由、缓存策略,都属于 harness 的"成本面"。

参见 LangChain anatomy 中关于 budget guard 的部分。

04
安全合规需要 harness

prompt injection、越权操作、PII 泄露 —— 这些靠 "model 自己注意" 是兜不住的。围栏、沙箱、审批流,必须在 harness 层强制。

参见 Thoughtworks · Birgitta Böckeler 的 trust boundary 论述。

05
评测可观测决定迭代速度

没有 trace、没有评测集,改 prompt 就是赌博。harness 的"检"那一层,决定团队每周能迭代几次、敢不敢上线。

参见 AGENTS.md 关于 eval-driven development 的约定。

04 · ANTI-PATTERNS

3 个反例:没有 harness 的代价

见过太多团队"接了模型 = 做了 AI 产品",上线一个月就被用户骂下架。原因往往就这三类。

ANTI-PATTERN · 01
没工具调用的 agent

用户:"帮我把这个 bug 修了。" Agent:"以下是建议步骤:1. ……" —— 然后就没了。

代价:用户要复制粘贴 5 次才能让事情发生,体验等同于 ChatGPT,凭什么收钱。

ANTI-PATTERN · 02
没评测的 agent

改了一版 prompt,产品经理在 Slack 里贴一句"感觉好多了!"就上线。三天后用户投诉:回答变啰嗦了。

代价:每次上线都是黑盒赌博,团队没法回答"它现在到底比上周好还是坏"。

ANTI-PATTERN · 03
没记忆的 agent

用户第 7 次告诉同一个 agent "我用 Python 不用 Java",每次对话都得从头解释,像金鱼。

代价:留存惨淡。用户感受到的不是"AI",是"PA 系统"(每次都要重填问卷)。

这三个反例的共同特征:把 AI 产品的工程缺陷,当成"模型还不够好"。事实上,它们都是 harness 缺位的问题,不是模型问题。

05 · WHY PM

为什么 PM 必须懂这个

这不是工程黑话,这是 PM 的新 baseline。

AI 产品经理过去三年的能力栈是: 用户洞察 + 产品定义 + 写 PRD。这套已经不够了。原因很简单 —— 当你的产品由一个 agent 提供主要价值,而你完全不懂这个 agent 是怎么被"包出来"的,你就没法做任何有意义的产品决策:工具该开几个、记忆多深、评测怎么设、什么动作必须人审、出错往哪兜……这些每一条都是产品决策,不是工程决策。

把 harness 纳入工程认知,意味着 PM 把"模型层"和"产品层"中间那一截 —— harness 层 —— 当成自己的设计对象。不是去写代码,而是能看懂 trace、能读 evals 报告、能跟工程师讨论"这个 retry 策略合不合理"、能在面试里讲清"我们家 agent 的 harness 设计"。

这是 2026 起,头部公司面 AI PM 时,实际会拷打的东西。OpenAI、Anthropic 自己的招聘 JD 里,已经在 PM 岗写"familiar with agent harness design"。

不懂 harness 的 AI PM,就像 2010 年不懂 RESTful API 的移动 PM。不是不能干,而是没法跟工程师"对等地讨论产品"。
