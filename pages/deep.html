03
解剖学 转·知·做·检 四部件深拆 —— 主循环、上下文、记忆、工具、执行边界、评测
6 个可视化
§1 · 开篇

打开 harness,
看里面的 6 个器官

板块① 你已经知道:Agent = Model + Harness。 这一板块,我们把 harness 的皮掀开,看里面到底有几个器官、各管什么。

Harness 存在的唯一理由,是补"模型做不到的事"。 所以"它有几个部件",其实是"模型有几种关键的'做不到'"。

按这条原则数下来,组件级 harness 有 6 个部件,挂在四个字上: 转 · 知 · 做 · 检。

§2 · 总览

整体地图:
转 · 知 · 做 · 检

5 + 1 结构:转 / 知 / 做(共 5 个部件)在 loop 里头干活,让 agent 跑起来 —— 是运行时部件。检(2 个部件)在 loop 外头,告诉你"跑得好不好" —— 是元层。

§3 · 转

主循环 ——
agent 的命根子

聊天机器人:模型跑一遍,出个答复,停。 Agent:模型跑一遍 → 看工具结果 → 再跑一遍 → 再看 → 再跑 ...... 这个"再"就是循环。没有这个循环,再强的模型也只是问答机。

反直觉 —— 用户说一句话(一个 turn),主循环可能转几百圈才停。OpenAI 在官方文章里说过:一个 turn 里 agent 可能发起数百次工具调用。 "一句话 ≠ 一步" 是 PM 的第一个反直觉。

PM 落点

⏱
这个 agent 最多自己跑多久?
💵
最多花多少钱?
🙋
卡住了停下问人、还是继续试?
🛑
用户能不能中途喊停?

这几个数,直接决定产品的成本、安全感和体验。

§4 · 知

上下文 + 记忆 ——
桌面和文件柜

记忆是源,上下文是容器。 每一轮,harness 像个秘书 ---- 从文件柜(记忆)、资料库(检索)和今天的对话(历史)里, 挑出该带的,装进桌面(上下文)。这个动作叫"上下文装配"。

PM 落点

✍
该记什么、不该记什么(写策略)
🗑
过期了怎么扔(衰减策略)
🔍
做哪些动作前主动查记忆(读触发)
🔒
谁能看谁不能看(scope / 隐私)

这几条,直接决定产品的连续感、智能感、合规风险。

§5 · 做

工具 + 执行边界 ——
让它能动手,又管得住

工具是 agent 的"手" ---- 读写文件、跑命令、调 API、发邮件 ...... 执行边界则管住这双手: 不是一根滑块,是三根独立的旋钮(彼此正交,各管各的,可任意组合)。

沙箱 vs Hook ---- 容易搞混的一对

🏰
沙箱
性质结构性、静态的环境约束
比喻城墙 / 假电脑
管什么根本能不能做到
适合大类、永久、物理可达性
强度硬 ---- 物理上没接口
🛡
Hook
性质临场、动态的动作拦截
比喻门口的卫兵
管什么这一次让不让做
适合按动作 / 内容 / 上下文临场判断
强度软 ---- 靠代码逻辑,代码错就漏

PM 落点

🏰
这个 agent 在哪几种环境跑?(沙箱档位)
👤
哪类动作必须问人?(审批策略)
🛡
业务有没有特殊规则要拦?(hook)
⨉
三根旋钮组合:沙箱兜底 + 审批兜底 + hook 兜底
§6 · 检

评测 + 可观测 ——
让你知道它跑得好不好

转 / 知 / 做 让 agent 跑起来;检在 loop 外头,负责告诉你"跑得好不好"。 它不让 agent 转一圈,但 ----

没评测的 harness 是裸奔;没可观测的 harness 是开车不看仪表盘。

PM 落点

🎯
上线前,有没有 golden set 跑一遍?(质量基线)
📊
在线 KPI 有没有 dashboard + 告警?(实时雷达)
🔎
用户报 bug,能不能 5 分钟内 replay?(debug 能力)
💰
出问题前,能定位"谁在烧钱"?(成本归因粒度)

任何一道答"没",产品就处于"还活着但看不见自己"的状态。出事故只是时间问题。

§7 · 分工地图

症状归器官 ——
PM 的听诊器

读完 6 个器官,最重要的 PM 能力是 ---- 看到一个症状,立即识别它归哪个器官。 多数事故不是"模型不行",是特定器官的设计缺失。

§8 · 总结
6 个器官,挂在 4 个字上:转(主循环)· 知(上下文 + 记忆)· 做(工具 + 执行边界)· 检(评测 + 可观测)。 前 5 个让 agent 跑起来,第 6 个让你知道它跑得好不好。 每一件,都是在补"模型当下还做不到的事"。

每个器官,落到 PM 都不是代码,是一组产品决策: agent 自己能跑多远?能记什么?能动什么?要不要问人?有没有自检? 回答这些问题,就是 PM 设计 agent 产品的工作。

§1 · 第一步

做 agent 产品的第一件事
不是选模型，是选尺度。

板块① 的"三个尺度"在这里第一次发挥用处。 选错尺度 = 白干。问 3 个问题就能定位 ——

🧩
COMPONENT
组件级

1 个 agent + 用户每次喊 + 单次任务

IDE 里的代码补全 / 嵌在 Office 的写作伴侣
🔄
TASK
任务级

多个 agent + 有终点的长任务

一句话造一个完整应用 / 一晚上整理一本书
🏢
ORGANIZATION
组织级

一支队伍 + 永远在演进的产品

公司内部 AI 编码平台 / agent 承包 80% 日常
选错尺度的代价 —— 用任务级 harness 解决一个组件级问题 = 用航母接客人。慢、贵、复杂度爆炸。
§2 · 分工

平台层 vs 应用层
你的活在应用层。

一个 harness 分两层。平台层已经有人做了，你不必造轮子。 认清"你能造什么"是方法论第一课。

最小 Harness 铁律
6 件全有，才叫 harness；
少一件，你做的是 demo。

§1 · 为什么

板块 ①-⑧ 都在讲价值。
这一节讲争议、局限、未解。

任何刚结晶的概念都会面对质疑。 能讲清质疑，才说明你不是"信徒"—— 比信徒值钱。

面试官最爱问： "你怎么看 harness 的未来？" "模型变强后还有 harness 吗？" —— 这两题答不好，前面 ①-⑧ 的火候全废。

核心立场 —— 把 harness 当工具，不当神药。 工具会迭代、会更新、会在某些场景被替代 —— 这很正常。 用工具的人不会因为工具升级而失业，"把工具当信仰"的人才会。

§7 · 启示与总结

给 PM 的
5 条实战启示。

把 harness 当工具，不当神药 —— 5 条 ——

Harness Engineering 不是噱头，但也不是终局。 它是过渡期的关键技术。 对 PM 来说，值得学的不是某个平台的 hook 语法， 是 harness 思维：把"agent 怎么跑"拆成零件去决策。 这种思维抗模型变化、跨平台通用，跟着你一辈子。
AI 产品的护城河，永远在"应用层" —— 项目知识、业务边界、评测体系。 机械那层会被模型吃掉，但应用层永远在，而且会越来越重要。 —— 如果你只能从这门课带走一句话，带走这句。
