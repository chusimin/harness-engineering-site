01 · WHAT IS IT

是什么

Harness Engineering（驾驭工程）= 包在模型外面、把被动模型变成能干活 Agent 的那一整圈代码与结构。

01 · DEFINITION

一句话讲清楚

Agent = Model + Harness。模型负责"想"，harness 负责让它"能干活"。

Agent = Model + Harness

这个公式的两位作者 —— Anthropic OpenAI —— 在 2026 年初先后撰文，正式定义了这个工程领域。

模型本身是无状态的、被动的、不能调用工具的。而一个能干活的 Agent，需要状态管理、工具调用、安全围栏、可观测性、评测体系 …… 这些"包在模型外面的代码与结构"，就是 harness。

02 · THREE SCALES

三个尺度

同心圆模型 —— 各家定义不冲突，只是站在"尺度尺"的不同刻度。

COMPONENT · 组件级
裹一次模型调用

最小单位。一次有围栏的模型调用 + 一次工具调用。像电商客服回答一个简单问题。

代表观点： LangChain

TASK · 任务级
编排多步完成一件事

中等粒度。多圈循环、多工具混用、边界判断随动作变化。像 Claude Code 修一个 bug。

代表观点： Thoughtworks

ORG · 组织级
管一支 agent 队伍

最大粒度。多 agent 路由协作、多层记忆、组织级策略、全局仪表盘。像一个客服中心管几十号人。

代表观点： Anthropic

组件级 ⊂ 任务级 ⊂ 组织级。 外层包裹内层 —— 做组织级 harness，必然也在做任务级、组件级。

03 · 6 COMPONENTS · ONE LOOP

六大组件 · 一轮循环

不管 harness 在哪个尺度，里面跑的都是这 6 个部件 —— 按顺序在一轮循环里走完。下面三个模拟器就是它在不同尺度的样子。

结果回流 · 下一圈 ↻
1
上下文管理
装桌面：system prompt + 对话历史 + 工具列表 → 模型此刻看见的一切
↓ 查记忆，补进桌面
2
记忆系统
从记忆库调取长期事实（用户偏好、历史决策…）补入上下文
↓ 桌面装好，交给模型
3
Agent 循环
模型思考，决定下一步：输出 tool call 或最终回答
↓ tool call 先过安检
4
执行边界
沙箱检查 → 审批策略 → Hook：放行 / 拦截 / 问人
↓ 放行后，真正执行
5
工具生态
调用 API / 读写文件 / 发消息… → 拿到结果
⑥ 评测 + 全程可观测 · Trace · 成本 · 告警

装桌面 → 补记忆 → 模型想 → 过安检 → 动手做 → 结果回流 → 再来一圈

任务没完，循环继续；任务完成或触发缰绳，循环停。

05 · NESTED LAYERS

嵌套关系

Harness 工程不是凭空出现的，它是从更小的实践一层层包出来的。

最内层
提示词工程 · Prompt Engineering
研究"怎么写好一条 prompt"
⊂
中间层
上下文工程 · Context Engineering
研究"给模型什么上下文"（含 RAG、记忆、工具）
⊂
最外层
Harness 工程 · Harness Engineering
研究"包出一个能跑、能管、能评的 Agent 产品"
