01 · ANTHROPIC

Claude Code

Anthropic 出品的终端编程 agent，2024 年发布。在 2026 年的官方文档里被定位为 harness 三个尺度全覆盖的工程载体。

形态：CLI 工具 模型：Claude Sonnet / Opus 尺度：任务级 + 组织级 发布：2024 年
EVOLUTION · 演化路线
1. 初代 · 终端补全 + ask 对话
最初只是 CLI 里跟 Claude 聊天，能读文件、写文件。
2. 引入 Sub-agents
并行思考多方案，把"探索"和"执行"拆开。
3. Plan mode 上线
长任务先出计划，人审节点强制存在 —— 任务级 harness 的关键拼图。
4. CLAUDE.md / AGENTS.md 项目记忆
每个仓库携带规则书。打开就有上下文，不用每次重述。
5. MCP 集成 · 标准化工具发现
从写死工具到对接 MCP server，组织级 harness 的雏形。
DESIGN HIGHLIGHTS · 设计亮点
长任务 harness sub-agents 并行探索 + plan mode 强制人审，能跑数小时不漂。
项目级记忆 CLAUDE.md / MEMORY.md 注入项目规则，跨会话延续。
危险命令 Hook rm -rf 等破坏性操作进入 hook 流程，强制确认。
工具组合 Edit / Bash / Read / Glob / Grep 五件套覆盖编程全场景。
最终形态：terminal-native, project-aware coding agent。装在哪儿就能跑，规则随仓库走，敢长任务。
PITFALLS · 踩过的坑
  • 长任务跑偏后难以重定向 —— 通过 plan mode 强制人审节点缓解
  • 危险命令早期靠用户警觉 —— 现在 hook 强制拦截破坏性操作
  • 多 sub-agent 并行时上下文成本高 —— 通过模型分层（Opus 规划 + Sonnet 执行）平衡

02 · OPENAI

Codex

OpenAI 在 2025-2026 年推出的云端独立 agent。"harness engineering"这个词，就是它们在 2026 年 2 月那篇博客里正式命名的。

形态：云端 Web / IDE 集成 模型：GPT-5 / o-series 尺度：任务级 发布：2025 年
EVOLUTION · 演化路线
1. Copilot Autocomplete 时代
最初的 Codex 是 GitHub Copilot 背后的补全模型 —— 还不算 agent。
2. ChatGPT Code Interpreter
第一次把模型 + 沙箱 + 工具组成完整闭环，雏形已现。
3. Codex Cloud Agent（2025）
独立云端工作环境，每个任务一个干净容器。
4. PR-style Output
输出标准化为 PR diff，让人审有抓手。
5. "Harness Engineering"正式命名（2026-02）
官方博客把这套外圈代码命名为"harness engineering"。
DESIGN HIGHLIGHTS · 设计亮点
全云端沙箱 每次任务一个干净容器，无副作用、可重放。
干净环境每次 没有"上次没清的状态"问题，对外部世界零污染。
PR diff 输出 所有改动收口到 PR 视图，人审节奏稳定。
时间预算 每个任务有明确的 wall-clock / token 预算，跑超就停。
最终形态：cloud-native autonomous coding agent，产物是 PR。让"派一个任务出去"变成像派工单一样自然。
PITFALLS · 踩过的坑
  • 容器冷启动慢 —— 高频小任务体感比 Cursor / Claude Code 重
  • 任务无法跨多容器协作 —— 边界即沙箱，复杂场景需要拆任务
  • 纯异步模式让人脱离感强 —— 加了 Slack / 邮件回执后才缓解

03 · ANYSPHERE

Cursor

Anysphere 出品的AI 原生 IDE。从 VS Code fork 出来，把 harness 钉死在编辑器里 —— 同屏可见、即刻反馈。

形态：IDE (VS Code fork) 模型：多模型可选（Claude / GPT / 自研） 尺度：组件级 发布：2023 年
EVOLUTION · 演化路线
1. VS Code Fork
从 VS Code 拉出一支，专门重写 AI 集成层。
2. Tab Completion
智能补全 + 多行预测，最初的"组件级 harness"。
3. @ Context References
@文件、@符号、@文档 —— 让上下文管理变成键盘动作。
4. Composer Agent Mode
多文件批量改动，从"裹一次调用"扩展到"裹一个小任务"。
5. Inline Lint + Diff Review
所有改动都进 diff 视图，人审是默认动作。
DESIGN HIGHLIGHTS · 设计亮点
仓库级索引 打开项目就自动建索引，@-context 秒级响应。
Composer 多文件改 一个意图，跨多个文件一致性修改。
@ Context 系统 文件、符号、文档、Web 都能 @，上下文管理用户主导。
Inline Lint 改完即刻显示 lint / 类型错误，反馈环最短。
最终形态：IDE-native agent IDE。把 harness 嵌进编辑器的每个键位，人和 agent 共享一个屏幕。
PITFALLS · 踩过的坑
  • 多文件改动一致性早期依赖人审 —— Composer 之前容易"改一处忘一处"
  • Context window 用满后效果断崖式下跌 —— 大仓库需要手动管理 @ 引用
  • IDE 形态注定难承长任务 —— 用户合上电脑就停了

04 · CODEIUM

Windsurf

Codeium 出品的"流模式"IDE。最独特的发明是 Flow Memories —— 让 agent 跨会话记住跟你的协作历史。

形态：IDE (Codeium IDE / 插件) 模型：多模型可选 + 自研 SWE-1 尺度：任务级（强记忆） 发布：2024 年
EVOLUTION · 演化路线
1. Codeium 补全（前身）
VS Code 等多 IDE 插件，对标 Copilot 的补全产品。
2. Windsurf IDE 发布
2024 年推出独立 IDE，重做整个 AI 集成层。
3. Cascade Flow 模式
把"补全→对话→多步执行"统一成一条流，长链条原生支持。
4. Flow Memories（核心创新）
跨会话持久记忆，agent 记得你的偏好和上周改过什么。
5. Write-Action Review
写操作（改文件、跑命令）默认进入 review，读操作直接放行。
DESIGN HIGHLIGHTS · 设计亮点
Cascade 流程模式 补全 / 对话 / 任务执行统一成流，链条原生支持长。
Flow Memories 跨会话 合上电脑明天再开，agent 还记得你们昨天到哪儿。
写操作 Review 读操作放行，写操作弹审 —— 边界按"破坏力"分级。
自研 SWE-1 模型 配合 harness 一起 co-design，模型层和外圈同步演化。
最终形态：long-flow IDE agent with persistent memory。区别于 Cursor 的"会话即生命周期"，Windsurf 让记忆跨越关机。
PITFALLS · 踩过的坑
  • Cascade 流程跨度长，需要明确"完成"信号 —— 否则容易越跑越远
  • Memories 累积污染：早期会"记住错的"，需要主动清理机制
  • "流"形态对新手有学习成本：不像 Cursor 那样所见即所得

05 · COGNITION AI

Devin

Cognition AI 出品的"AI 软件工程师"。2024 年那段震撼演示视频后，第一个真把"agent 像同事一样在 Slack 收到工单"做成产品的团队。

形态：云端工作台 模型：多模型混合 尺度：组织级雏形 发布：2024-03
EVOLUTION · 演化路线
1. 概念演示（2024-03）
震撼视频：Shell + 浏览器 + 编辑器三件套，一镜到底跑 Upwork 任务。
2. 公测开放
从 demo 到付费产品，第一批用户开始压测真实场景。
3. 团队版（Team）
多用户共享同一个 Devin，权限 / 审计 / 计费分层。
4. Slack / Linear 集成
从 web 工作台外溢到工作流：Slack 派单、Linear 跟进。
5. 实时回放（Playback）
每一步动作录像回放，可观测性是组织级 harness 的入场券。
DESIGN HIGHLIGHTS · 设计亮点
完整工程师工作台 Shell + Browser + Editor，跟人类用的一模一样。
长任务 1-8 小时 单次任务可跑大半天，是当前公开产品里最长的。
Slack 通信 派单、追问、确认都在 Slack 里，agent 像异步同事。
实时回放 录像 + trace，事后可审，是组织级可观测的基础。
最终形态：cloud-based autonomous software engineer。不是助手、不是工具 —— 是产品定位上的"一名独立工程师"。
PITFALLS · 踩过的坑
  • 早期演示效果难复现 —— 真实任务的成功率远低于 demo 暗示的水平
  • 长任务漂移：跑到第 5、6 小时容易钻牛角尖，需要中段强制人审
  • 公开评测（SWE-Bench）的成绩有过争议 —— 用户对"自主"的期待和现实有差距
  • "AI 工程师"叙事吸引面试官，但落到团队里更像是"高级实习生"

07 · PM TAKEAWAY

PM 选型建议

你做的产品落在哪个尺度，就去抄那个尺度上做得最好的厂商的作业。

1
COMPONENT · 组件级
你做：in-app 助手、简单问答、SaaS 嵌入

学 Cursor 的 IDE 深度集成思路：同屏、即刻反馈、@-context、用户主导上下文。

关键能力：仓库级索引、@ 引用系统、diff 同屏。

2
TASK · 任务级
你做：自动化工单、bug 修复、独立任务

学 Claude Code 的 plan mode + 工具组合，或 Codex 的云端沙箱模式。

关键能力：plan / sub-agents、强工具组合、PR-style 输出、人审节点。

3
ORG · 组织级
你做：多 agent 队伍、长期产品、平台

学 Devin 的工作台思路 + Anthropic 的治理框架。

关键能力：实时回放、Slack 集成、分层记忆、组织审计、长任务断点。

没有完美的 harness，只有匹配你产品阶段的 harness。 选型不看技术多酷，看你的用户在哪个尺度上需要 agent —— 用户在 IDE 里就抄 Cursor，用户在工单系统里就抄 Claude Code，用户要一名"虚拟同事"就抄 Devin。
