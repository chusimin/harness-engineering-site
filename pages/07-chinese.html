§1 · 时间差

中文圈和英文圈，
差在哪？

Harness Engineering 在 2026 年初英文圈引爆。到目前为止， 英文圈已经把"概念 + 实践"都讲透了， 中文圈有零散好文章，但还没有成体系的 PM 视角中文教程。

§3 · SDD 争论

SDD —— 中文圈
最大的方法论争论

SDD = Spec-Driven Development = 规范驱动开发。 核心主张：在让 AI 写代码之前，先写一份完整的 spec（需求、接口、数据结构、测试用例）， AI 按 spec "翻译"成代码。

SDD 想解决的痛点 —— "AI 输出不可控，需要更强约束" —— 是真实的。 但争论在于：它跟 harness 是什么关系？

谁在推 SDD？ 主要两股力量：❶ 阿里（通义灵码 + 白皮书 + 培训 + 大厂渠道）—— 中文圈最大推手。 ❷ GitHub Spec-Kit（社区开源）—— 西方社区也有，影响力相对中性。 中国大厂的 to-G / 国央企业务特别愿意采纳 SDD：因为 spec 可留痕、可审计、可签合同。

批评派的三个核心声音

中文 PM / 工程圈对 SDD 的常见批评 ——

  1. "为中层经理设计，不是为一线工程师设计" —— 让管理层有控制感，但跟一线产出力没强相关。
  2. "瀑布开发最大的问题被包装成 AI 时代的革新" —— 过度前置规划在信息不全时做最多决策，敏捷早已证明这是错的。
  3. "AI 最大红利是迭代成本接近零，SDD 反而把重文档的痛苦塞回来" —— 改代码这么便宜，为什么还先写 50 页 spec？

这些批评有道理。但前面的场景卡片也说了 —— 要看场景。 二元站队是错的。

§4 · 上下文工程

上下文工程在中文圈

一句话总结

中文圈对 harness 的讨论还没收敛 —— 术语三种用法、SDD 争论白热化、厂商各异、上下文工程过热但缺方法论。

作为 AI PM，你的位置不是站队，是搞清场景： 消费级 / 创业期默认 harness 轻量；to-G / 合规重场景 SDD 有真实价值；两者不是对立，是光谱。

能在中文求职市场系统化地讲清这些，你就比 80% 的候选人更专业。
