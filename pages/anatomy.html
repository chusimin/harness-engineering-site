03 · ANATOMY

解剖学

拆开一个 harness，看里面每个部件的职责。

01 · OVERVIEW

转 · 知 · 做 · 检

不论组件级、任务级还是组织级，一个 harness 都可以拆成这四个部件。谁缺谁补、谁弱谁砸钱。

⟳
转 · LOOP

主循环 —— 决定 Agent 下一步做什么。心跳与节拍器。

◉
知 · MEMORY

上下文与记忆 —— Agent 知道什么、记得什么。工作台与抽屉。

⚡
做 · TOOLS

工具与执行 —— Agent 能动什么、不能动什么。手脚与边界。

◈
检 · EVAL

评测与可观测 —— 跑得好不好怎么知道。仪表盘与放大镜。

这四个部件，在 组件级（裹一次模型调用）、任务级（编排多 Agent）、组织级（罩一支队伍）里都一一对应 —— 只是尺度变了，循环、记忆、工具、评测的颗粒度跟着变大。 LangChain 把这四件事称作 "the anatomy of an agent harness"。

02 · 转 · LOOP

主循环决定 Agent 每一步做什么

模型本身是无状态的，跑完一次就结束。让它"持续工作"的，是外面那个 while。循环的形状，决定 Agent 的人格。

PATTERN · 01
ReAct

Think → Act → Observe，每一步都写出推理。模型把"想"暴露在 trace 里，可调试、可干预。
典型用法：问答型 Agent、客服机器人、单步工具调用。

PATTERN · 02
Plan-and-Execute

先做完整 plan，再分步执行。计划阶段一次性消耗 token，执行阶段每步只看子任务上下文。
典型用法：多步骤代码任务（Claude Code、Devin）、研究型 Agent。

PATTERN · 03
Adaptive

不固定循环结构，由模型动态判断"现在要 think、要 act、还是要请用户介入"。最贴近 Claude 4.x、GPT-5 这类 reasoning model 的实际形态。
典型用法：长程任务、深度研究、复杂工程。

循环的终止条件

循环不能无限转。哪一条先成立，哪一条先停。常见的终止条件：

  • 最大轮次（max_steps / max_iterations）—— 防止死循环烧 token。
  • 成功判据（success criteria）—— 由模型自报"任务完成"，或由外部验证器（如测试通过）触发。
  • 用户中断（user interrupt）—— 把方向盘交回人手里，常见于 human-in-the-loop。
  • 预算耗尽（budget cap）—— token 数、调用次数、时间预算任一耗尽就停。

ReAct 这套思路的起点是 2022 年 Yao 等人的论文 ReAct: Synergizing Reasoning and Acting in Language Models， 后来被 Anthropic 2026 年的 harness 设计文 扩展成"长程任务里循环 + checkpoint + 子 agent"的复合范式。

03 · 知 · MEMORY

Agent 的工作记忆与长期知识

模型只看得到 context window 里的 token。放进窗口的，是它的"全部已知世界"。

SHORT-TERM · 短期记忆
这一次任务里的工作台

单次对话/单次运行的状态，跟着任务结束就丢。

  • 对话历史窗口（messages 数组）
  • 思维链 / scratchpad（模型自己写给自己看的草稿）
  • 工作变量（中间结果、文件路径、当前 step 编号）
LONG-TERM · 长期记忆
跨次跨用户的抽屉

跨任务存活的知识，按需召回。

  • 向量库 RAG（按语义检索文档片段）
  • 知识图谱（结构化实体与关系）
  • 用户偏好 / 历史摘要（"我喜欢用 TypeScript"）

上下文窗口管理

就算用上百万 token 的窗口，也会撞墙。所以 harness 必须主动管 context：

  • 摘要压缩（summarization）—— 当对话变长，把早期回合压成一段摘要，腾出位置给新内容。
  • 滑动窗口（sliding window）—— 只保留最近 N 个 turn，老的丢弃。
  • RAG 注入（retrieval-augmented generation）—— 不把全部知识塞进 prompt，而是按当前 query 检索最相关的片段拼进去。
  • 记忆固化（memory consolidation）—— 任务结束后，把"值得记的东西"提炼成长期记忆条目，下次再取。

LangChain 把这一层称作 Agent 的 "memory system"，并主张短期与长期分层、不要混在一个 store 里。

04 · 做 · TOOLS

工具是 Agent 的手脚，权限是边界

模型不能动你的文件、不能上网、不能跑代码 —— 除非你把这些能力以"工具"的形式注册给它。注册一个工具，就是开一条口子。

CATEGORY · 01
文件系统

read / write / list / search。Agent 在工作目录里读写文件，是绝大多数 coding agent 的基础能力。
示例：Read、Write、Edit、Grep、Glob。

CATEGORY · 02
代码执行

shell 命令、Python REPL、容器内 build/test。要在沙箱里跑，不能直接接到生产环境。
示例：Bash、code_interpreter、sandboxed Docker。

CATEGORY · 03
网络与 API

HTTP fetch、搜索引擎、内部业务 API、MCP servers。决定了 Agent 能不能"出门办事"。
示例：WebSearch、WebFetch、内部 CRM API、MCP tools。

工具 schema 怎么定

每个工具都需要一个机器可读的 schema，告诉模型：这工具叫什么、做什么、参数是什么、返回什么。一份合格的 schema 至少包含：

  • 函数签名 —— name、description、parameters（JSON Schema 格式）。
  • 参数验证 —— 模型生成的参数先经类型/范围校验，挡住"幻觉调用"。
  • 沙箱化 —— 危险操作（写文件、跑代码、调用付费 API）必须跑在隔离环境里。
  • 权限分层 —— 读操作默认放行；写/删/付费操作走 human-in-the-loop 审批。

Human-in-the-Loop Checkpoint

不是所有动作都该让 Agent 自动执行。"不可逆动作前必须人审"是社区共识 —— OpenAI 的 Codex harness 把这条做成默认行为：删文件、发邮件、提 PR 前都要人点确认。 Anthropic 长程 harness 则进一步把权限按"低/中/高风险"分了 3 个等级。

05 · 检 · EVAL

没有评测，就是黑盒

Agent 跑赢了没？换了 prompt 是好了还是坏了？没数据，全靠"感觉"——那不叫工程。eval 是 harness 工程师的护身符。

METHOD · 01
离线评测集

一组固定的"题目 + 标准答案"。每次改代码、换模型、调 prompt，都跑一遍，看分数变化。
代表：SWE-bench、GAIA、内部业务 golden set。

METHOD · 02
在线 A/B 实验

线上分流，新旧 harness 各跑一部分真实流量。看任务完成率、用户满意度、续费率的差异。
代表：Cursor 灰度发布、Claude Code 渠道实验。

METHOD · 03
Trace 链路

每一次 Agent 运行都记录完整 trace（每一步 input/output/工具调用/耗时）。出错时回放，平时做归因。
代表：LangSmith、Weights & Biases、自建 OTel + ClickHouse。

关键指标

不论用哪种方法，最终落到几个数：

  • 任务完成率（task success rate）—— 端到端跑通的比例。最重要的"北极星"。
  • Token 成本（cost per task）—— 一次任务平均烧多少钱。商业化的命脉。
  • 平均轮次（avg steps）—— 一次任务走了多少步循环。越少越高效，但也要警惕"走得少是因为放弃了"。
  • 人工介入率（human handoff rate）—— 多少比例的任务最终求救于人。衡量自动化深度。

评测能力的方法学，可以看 METR 的 capability elicitation 指南 —— 它强调一件事：你测出来的能力，永远是你给的 harness 的下限。换个更好的 harness，分数能涨。

06 · DESIGN PATTERNS

6 种常见设计模式

把上面四件事（转·知·做·检）以不同方式组合，得到这 6 种已经在工业界落地的 harness 形态。挑一个起手，再按需混合。

PATTERN · 01
Single-Agent Loop

最朴素的形态：一个 Agent + 一个 ReAct 循环 + 一组工具。

WHEN · 任务边界清晰、可拆步骤少。
⚠ 走偏了没人拉它，长任务容易跑飞。
PATTERN · 02
Multi-Agent Orchestration

一个 Orchestrator 派活给多个 Worker Agent，并行或串行收尾。

WHEN · 任务可分工，子任务之间相对独立。
⚠ 通信开销 + 难调试，token 成本爆炸常见。
PATTERN · 03
Human-in-the-Loop Checkpoint

不可逆动作前停下，等用户点确认或修改后再继续。

WHEN · 产线、金融、运维等高风险场景。
⚠ 设错检查点用户会疲劳，设少了出事故。
PATTERN · 04
Reflection / Self-Critique

一个 Agent 跑完，再叫另一个（或同一个换 prompt）的 Agent 来挑刺，循环直到通过。

WHEN · 输出质量要求高、容错小。
⚠ 自反容易过度自信、循环死锁。
PATTERN · 05
Tool-Augmented Retrieval (RAG)

每次推理前先去向量库 / 搜索引擎拿事实，再让模型基于事实回答。

WHEN · 知识更新快、有内部文档要查。
⚠ 检索质量差时，幻觉比不带 RAG 还多。
PATTERN · 06
Sandboxed Code Execution

在隔离容器/微 VM 里跑 Agent 生成的代码，把结果回填给模型继续推理。

WHEN · 数据分析、coding agent、动作可逆。
⚠ 沙箱逃逸、资源占用、状态污染都常踩坑。
