01 · CONTEXT MANAGEMENT

上下文管理 · 装桌面

把模型"此刻能看见的一切"组装好。

把模型"此刻能看见的一切"组装好。包括 system prompt、对话历史、当前任务、可用工具 —— 这一切构成了模型决策的全部依据。

模型每一次推理都是"无状态"的：它不记得上一轮发生了什么，所有状态都得在这一轮一次性塞进上下文窗口。上下文管理就是这个"塞"的工程。

子机制 · SUB-MECHANISMS
M1 · SYSTEM PROMPT
System prompt 装载

角色定义、行为约束、输出格式。是 agent 的"人格说明书"。

M2 · HISTORY
对话历史拼装

用户消息、模型回答、工具结果按时序回流到上下文。

M3 · TOOL LIST
工具列表注入

当前可用工具的 schema 描述，告诉模型"你能干什么"。

M4 · BUDGET
Token 预算切割

窗口管理。超出走摘要 / RAG / 截断。预算就是钱。

M5 · PRIORITY
指令优先级

系统级 > 项目级 > 用户级。冲突时上层赢。

M6 · MULTIMODAL
多模态拼接

图片、文件、代码块按规范注入到上下文窗口。

CASE 电商客服查耳机 · 看上下文管理在干什么
→
装入 system prompt："你是 XX 品牌客服，语气亲切，不准透露内部系统名"
→
拼接用户消息："我上周买的耳机到哪了？"
→
注入工具列表：read_order, query_logistics, send_message
→
预算切割：token 预算 8K，历史消息保留最近 10 条，超出走摘要
✓
→ 这就是模型"此刻看见的一切"，模型据此决定下一步
PM 该问什么
  • Token 预算多少？历史从哪截断？溢出时走摘要还是 RAG？
  • System prompt 怎么管理（hardcode / 配置 / 版本化 / A/B 实验）？
  • 工具列表是动态的还是固定的？什么场景下隐藏哪些工具？
  • 多模态资源（图片、文件）怎么压缩、何时丢弃？

参考： AGENTS.md Anthropic harness design

02 · MEMORY SYSTEM

记忆系统 · 补背景

把长期事实补进上下文，让 agent 不像"金鱼"，每次对话都从零开始。

把长期事实补进上下文 —— 包括用户偏好、历史决策、组织知识。让 agent 不像"金鱼"，每次对话都从零开始。

记忆系统是"窗口之外的状态"：当前 token 窗口装不下的，都进记忆库；下一轮再按需调出来塞回上下文。

子机制 · SUB-MECHANISMS
M1 · SHORT-TERM
短期记忆

对话窗口、工作变量、当前 trace。一次会话结束就丢。

M2 · LONG-TERM
长期记忆

向量库 RAG、知识图谱、文档检索。跨会话存活。

M3 · LAYERED
多层记忆

用户级（个人偏好）/ 组织级（共享知识）/ 任务级（当前 ticket）。

M4 · CONSOLIDATE
记忆固化

什么时候把短期写回长期。需要触发器、去重、冲突合并。

M5 · RETRIEVAL
记忆检索

语义搜索 / 关键词 / 时间窗口 / 元数据过滤。

M6 · FORGET
遗忘策略

TTL、相关性衰减、用户主动删除。隐私合规也在这层。

CASE 客服查记忆库 · 看记忆系统在干什么
→
用户级检索：用 user_id 查向量库，命中标签 VIP / 上周投诉过物流慢
→
组织级检索：拉取该品牌的售后政策（VIP 7 天退、物流延迟可补偿）
→
任务级注入：把这次会话里抓到的关键字（订单号、商品型号）写入短期记忆
→
预算控制：检索到 20 条，按相关性分数取 Top 5 注入上下文
✓
→ 模型决定语气更小心、优先给解决方案、主动提补偿券
PM 该问什么
  • 哪些事实进短期、哪些进长期？固化触发器是什么？
  • 记忆多久过期？用户能否查看 / 删除自己的记忆（GDPR / 个保法）？
  • 多用户共享记忆时，权限怎么隔离？
  • 检索回来一堆记忆，怎么排序、怎么避免污染上下文？

参考： LangChain · The Anatomy of an Agent Harness Anthropic harness design

03 · AGENT LOOP

Agent 循环 · 模型想

agent 的"心跳"，决定下一步做什么、何时停。

模型思考 → 决策 → 工具调用 → 观察 → 再思考的主循环。这是 agent 的"心跳"，决定下一步做什么、何时停。

没有循环，模型只是个一问一答的聊天机器人。循环 + 工具 + 终止条件，才让它成为能干活的 agent。

子机制 · SUB-MECHANISMS
M1 · REACT
ReAct 模式

Thought → Action → Observation 三段式。最经典的循环结构。

M2 · PLAN
Plan-and-Execute

先列计划，再分步执行。适合任务路径明确的场景。

M3 · REFLECT
自反思 Reflection

走错路时主动回退、换方案。让 agent 不死磕在一条路上。

M4 · STOP
循环停止条件

任务完成、达到最大轮次、用户中断、触发缰绳。一定要有。

M5 · DISPATCH
子 agent 派遣

主 agent 调度子 agent，并行 / 串行 / 投票汇总。

M6 · ROUTE
多模型路由

简单任务用 Haiku，复杂用 Opus。成本 / 时延的工程取舍。

CASE Claude Code 修 bug · 看 Agent 循环在干什么
1
Thought：用户说登录页报 500 → 先看错误日志
2
Action：调用 read_file('logs/error.log') → 拿到堆栈
3
Thought：堆栈指向 auth.js:42 → 读那个文件
4
Action：调用 read_file('auth.js') → 看到空指针
5
Action：调用 edit_file 加判空 → 改完
6
Action：调用 run_tests → 全绿
✓
→ 满足停止条件（测试通过），循环结束。约 6 圈
PM 该问什么
  • 单轮最长几步？跑飞了用什么兜底（max_iterations、超时、token 上限）？
  • 什么算"任务完成"？是模型自己说完成，还是有外部判断器？
  • 走错路怎么纠？反思机制怎么触发？
  • 什么时候派子 agent、什么时候用便宜的小模型？

参考： ReAct paper (arXiv:2210.03629) Anthropic harness design

04 · EXECUTION BOUNDARY

执行边界 · 过安检

决定 agent 能干什么、不能干什么。

在工具真正执行前的最后一道安检。沙箱、审批、Hook、人机交接 —— 决定 agent 能干什么、不能干什么。

模型可能犯任何错；边界是"哪怕模型犯错，损失也可控"的工程兜底。这层做不好，前面 5 层都白搭。

子机制 · SUB-MECHANISMS
M1 · SANDBOX
沙箱隔离

容器、虚拟机、只读文件系统、网络白名单。把爆炸半径锁死。

M2 · APPROVAL
审批策略

高风险动作弹窗问人、金额阈值升级、敏感操作走二审。

M3 · PRE-HOOK
前置 Hook

工具调用前的检查器：参数过滤、敏感字段屏蔽、改写。

M4 · POST-HOOK
后置 Hook

工具调用后的检查器：输出审查、合规过滤、敏感信息脱敏。

M5 · ESCALATE
人机交接条件

不确定 / 高风险 / 用户情绪激动时主动 escalate 给人工。

M6 · ROLLBACK
回滚机制

错误操作的 undo / git revert / 数据库事务回滚。

CASE 两个真实拦截 · 看执行边界在干什么
A1
Coding agent 要跑 rm -rf / → 沙箱里的容器拦截
A2
审批策略弹窗："这个命令将删除根目录所有文件，是否继续？"
A3
用户拒绝 → 阻断，工具调用失败，结果回流到模型
B1
客服 agent 要承诺超额补偿 ¥5000（策略上限 ¥500）
B2
后置 Hook 拦截：检测金额超阈值 → 改写回复 / escalate 给主管
✓
→ 模型可能犯错，但损失被边界锁在可控范围
PM 该问什么
  • 哪些操作必须人审？审批走什么流程？审批人不在线怎么办？
  • 误操作如何回滚？回滚的粒度有多细（一条命令 / 一整次会话）？
  • 沙箱的爆炸半径多大？最坏情况能损失什么？
  • 升级到人工的阈值是什么？怎么避免"什么都问人"的躺平？

参考： OpenAI · Harness Engineering Martin Fowler · Thoughtworks

05 · TOOL ECOSYSTEM

工具生态 · 动手

agent 能"动手做"的全部能力。

agent 能"动手做"的全部能力。每个工具是一个有 schema 的函数，agent 通过 tool call 调用。

模型本身不能读文件、不能发请求、不能改数据。工具就是模型的"手"，工具生态的丰俭决定了 agent 的能力边界。

子机制 · SUB-MECHANISMS
M1 · SCHEMA
工具 schema 定义

函数签名、参数描述、返回类型。模型靠 schema 决定调谁。

M2 · DISCOVERY
工具发现

MCP 标准、动态加载、按场景启用 / 隐藏。

M3 · PERMISSION
权限分配

不同角色 agent 能看到不同工具集。最小权限原则。

M4 · COMPOSE
工具组合

chain of tools、子任务复用、宏命令封装。

M5 · ERROR
错误处理

重试、降级、超时、熔断。工具失败不能让 agent 卡死。

M6 · IDEMPOTENT
幂等性设计

工具调用可重复，避免副作用累加（重发邮件 / 重复扣款）。

CASE Claude Code 工具集 · 看工具生态在干什么
→
读类工具：Read / Glob / Grep → 权限低、无副作用、可任意调
→
写类工具：Edit / Write → 修改文件，需要走文件备份 / git 兜底
→
执行类工具：Bash → 权限最高、副作用大、要过更严的边界
→
schema 写得好：每个工具的参数描述能让模型一眼看懂"什么时候用我"
→
组合：Glob 找文件 → Read 读内容 → Grep 定位 → Edit 改一行（高频链路）
✓
→ 工具粒度 + 权限 + schema 质量 决定了 agent 好不好用
PM 该问什么
  • 工具粒度怎么定？大而全 vs 小而专，哪种更适合当前场景？
  • schema 描述质量怎么测？模型能不能一眼看懂"什么时候用我"？
  • 幂等性谁来保证？工具自身、还是 harness 层去重？
  • 哪些工具组合是高频链路？要不要封装成宏命令？

参考： Model Context Protocol (MCP) OpenAI · Harness Engineering

06 · EVAL & OBSERVABILITY

评测 + 可观测 · 全程记账

让 agent 从"黑盒"变成"可控的工程系统"。

全程记账 + 长期评估。trace 链路、评测集、A/B 实验、告警 —— 让 agent 从"黑盒"变成"可控的工程系统"。

这是 harness 工程区别于"调 prompt 调出感觉"的关键。没有评测 + 观测，所谓"agent 变好了"就是玄学。

子机制 · SUB-MECHANISMS
M1 · TRACE
Trace 链路追踪

一个请求串起的全部步骤，含 token / 耗时 / 工具调用 / 结果。

M2 · GOLDEN SET
离线评测集

golden set、回归用例、benchmark。每次发布前跑一遍。

M3 · A/B
在线 A/B 实验

新 prompt / 新工具 / 新模型的灰度发布与对照。

M4 · METRICS
关键指标

任务完成率、平均轮次、token 成本、人工介入率、用户满意度。

M5 · ALERT
异常告警

失败率突增、token 用量超预算、延迟尖刺 → 实时报警。

M6 · COST
成本归因

按团队 / 功能 / 用户等级拆分。一笔账算到底。

CASE 客服 trace 复盘 · 看评测+可观测在干什么
→
一条 trace：路由 agent → 退货专员 → 投诉专员 → 人工升级
→
同一 trace ID 串起所有步骤：每一步的输入、模型决策、工具调用、耗时
→
事后复盘："为什么这单走了人工？" → 看 trace：在第 4 圈触发了金额阈值
→
评测集回归：这类 case 加进 golden set，下次模型升级前必跑
→
仪表盘：今日任务完成率 92%、平均 3.4 轮、人工介入率 7%、单 case 成本 ¥0.18
✓
→ 从"感觉 agent 还行"升级为"有数据 / 有趋势 / 有回归"
PM 该问什么
  • 评测集多大？覆盖了哪些场景？多久更新一次？
  • 关键指标是什么？怎么发现回归（昨天还好今天怎么变差了）？
  • 预警阈值在哪？谁来响应？响应 SLA 是多少？
  • 成本怎么归因？哪个功能最烧钱、能不能换便宜模型？

参考： METR · Capability Elicitation LangChain · Anatomy of an Agent Harness
