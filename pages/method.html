05 · METHODOLOGY

方法论

PM 怎么参与 harness 工程：4 个步骤 + 可 fork 的模板仓库。

01 · PM MINDSET

harness 不只是工程师的事

PM 不是写代码的人，但 harness 决定产品成败 —— PM 必须能读懂、能评审、能驱动。

01 · KNOW
PM 该懂什么

· 主循环（Loop）如何决定步数与停止条件
· 上下文窗口、记忆、RAG 的边界
· 工具 schema、权限、沙箱模型
· trace / span / 评测集是什么
· 成本 = Token × 轮次 × 调用量

一手参考： Anthropic

02 · ASK
PM 该提什么问题

· 这个 Agent 在什么条件下应该停？
· 失败时是回滚、重试、还是叫人？
· 上线后我看什么指标判断"还能跑"？
· 边界外的输入怎么拒绝？
· 每次调用大概花多少 token？

一手参考： OpenAI

03 · DECIDE
PM 该决策什么

· 哪些任务交给 Agent、哪些留给人
· 工具集的"必有 / 可选 / 禁用"
· 评测标准（什么算"通过"）
· 人机交接的触发条件
· 上线灰度策略与回滚阈值

一手参考： Thoughtworks

PM 工程素养 = AI 时代的核心能力。 不写代码 ≠ 不懂代码。在 harness 工程里，PM 要能跟工程师 用同一套术语 讨论上下文构造、工具边界、评测指标 —— 否则你定的 PRD 工程师没法落地，工程师写的 trace 你也看不懂。

02 · STEP ONE

第一步：定义 Agent 边界

哪些任务交给 Agent，哪些留给人 —— 这是 harness 设计的第一刀。

✓ FIT
适合给 Agent 的任务

· 高度结构化 —— 流程清晰、输入输出可形式化
· 有明确成功标准 —— 跑完后能判断对错
· 容错空间大 —— 错了能改、不致命
· 可拆分 —— 大任务能拆成可验证的子步
· 单步可验证 —— 每一步都有信号判断进度

✗ MISFIT
不适合的任务

· 强主观判断 —— 需要审美、价值观、共情
· 不可逆动作 —— 删库、转账、发邮件
· 高合规风险 —— 医疗、法律、金融决策
· 低样本量 —— 没数据训练、没数据评测
· 要求人际信任 —— 谈判、安慰、敏感对话

人机交接条件（Handoff Criteria）

Agent 能跑不等于该跑到底。当下面任何一个触发，应该停下来叫人：

  • 不确定性超阈值：模型自评 confidence 低于 0.6，或多次尝试结果不一致
  • 动作进入高风险区：涉及金额、用户隐私、外部 API 写操作、不可撤销动作
  • 预算/时间超限：单任务 token 超预算、循环超过最大轮次、单步耗时过长
  • 用户明确要求：用户说"我自己来"、"先停"、"等我确认"
  • 评测信号变化：在线指标突降、错误率上升，应触发降级到人工

参考： Anthropic Harness Design 把"何时叫人"列为 long-running agent 设计的核心约束之一。

03 · STEP TWO

第二步：设计工具集

工具是 Agent 的手脚，schema 是它的说明书。工具选不好，模型再聪明也干不成事。

⚒
工具选型清单

· 这个动作非工具不可吗？
· 有没有更小粒度的工具能替代？
· 工具是否原子化、可组合？
· 同一类工具是否有 N 个重复？合并
· 高频工具是否已缓存？低频是否懒加载？

{ }
Schema 规范

· 工具名是动词 + 名词，不超过 3 词
· description 写"什么时候用 / 不该用"
· 参数严格类型化，必填/可选明确
· 返回值结构稳定，error 单独通道
· 给一个 example 调用 + 一个反例

⌬
权限边界

· 读 / 写 / 执行 分级
· 单次调用预算（rows、tokens、$）
· 黑名单（哪些资源永远不能碰）
· 沙箱隔离（容器、网络、文件系统）
· 审计日志（谁、何时、调用了什么）

PM 审 tool spec 的 checklist

每一个工具上线前，PM 应该问工程师：

  • 输入输出明确？ 入参类型、必填项、输出 schema、错误码 —— 都有文档
  • 错误处理？ 失败时返回什么？模型能从错误信息里读出怎么修正吗？
  • 幂等性？ 同一输入调多次结果是否一样？写操作有没有去重 key？
  • 权限范围？ 默认最小权限，需要写权限的工具必须显式 opt-in
  • 可观测？ 每次调用是否能打 trace？延迟、错误率能不能聚合看？

参考： OpenAI Function Calling Docs 对 schema 写法有详细约束 —— 一个糟糕的工具描述，会让模型反复试错、烧 token。

04 · STEP THREE

第三步：构建评测体系

没评测就没迭代。不能数字化的能力，就不能持续优化。

OFFLINE · 离线评测
在数据集上验证

· 评测集构建：典型样本 + 长尾 + 反例。100-1000 条够用
· Benchmark 选择：SWE-bench、GAIA、TAU-bench 等公开集做参考
· 回归测试：每次改 prompt / 工具，先跑回归集再上线
· 人工标注：争议样本走双盲打分

ONLINE · 在线评测
在真实流量里验证

· A/B 实验：新版本 vs 旧版本 分流，看核心指标
· 用户反馈：👍/👎、修改率、放弃率、复访率
· 自动监控：异常请求、长尾延迟、错误聚类
· 影子模式：新版本在线跑但不出结果，对比差异

METRIC · 01
任务完成率

为什么重要：北极星指标，业务价值的最终衡量。

怎么读：分场景看（简单/复杂任务分桶），看绝对值和环比。突降 5% 是事故。

METRIC · 02
Token 成本

为什么重要：决定毛利。Token 浪费 = 钱包漏水。

怎么读：看 P50/P95，看每任务成本而非总成本。成本上升要看是任务变难还是有 bug。

METRIC · 03
平均轮次

为什么重要：反映 Agent 效率。轮次越多越烧 token、越慢。

怎么读：不是越少越好。看分布。长尾任务卡满 max_turns 是设计问题。

METRIC · 04
人工介入率

为什么重要：衡量 Agent 自主性。介入率越低，规模化越省人。

怎么读：不是越低越好。该叫人时不叫是事故。看"介入后用户修改率"判断是否过度自动化。

参考： METR Capability Elicitation 提供了一套行业级的 agent 评测方法论 —— 从评测集设计到能力上限的探测。

05 · STEP FOUR

第四步：治理与可观测

长期运行 = 持续治理。不能盯着、不能改、不能止损的系统不是产品，是定时炸弹。

◎
可观测

trace 链路：每次调用一条 trace，含 prompt、工具调用、结果、token 数

分级日志：DEBUG / INFO / WARN / ERROR，结构化输出

指标看板：实时 QPS、错误率、P95 延迟、成本流速

◆
安全

围栏：输入过滤（prompt injection）+ 输出过滤（PII、违规内容）

审计：高风险动作留痕，可追溯到具体调用

沙箱：代码执行隔离、网络白名单、文件系统只读

$
成本

预算：按租户、任务、工具分账，硬上限熔断

限流：QPS 限流、并发限制、token 速率限制

优化：prompt 缓存、模型分级（贵的留给硬骨头）

Agent 失败的事故响应

AI Agent 不像传统软件 —— 它的失败常常是渐进式、概率性、跨任务相关的。一次事故响应流程应该包括：

  • 检测：监控指标（成功率、延迟、成本）触发告警阈值
  • 降级：自动切换到更简单的模型或工作流，必要时全部降级到人工
  • 定位：通过 trace 还原最近 N 次失败调用的共性 —— 是 prompt 变了、工具坏了、数据偏移了？
  • 回滚：保留至少 2-3 个历史版本（prompt / 工具 schema / 模型版本），可一键回滚
  • 复盘：事后总结，加入回归集，防止下次再犯

参考： LangChain · The Anatomy of an Agent Harness 把"持续运营"列为 harness 三大支柱之一，与 loop、tool 并列。

实战模板仓库

4 个可 fork 的 PM 模板 —— 从想法到上线，每一阶段都有结构化产出。

TEMPLATE · 01
BRD 模板 · Business Requirement

判断"这个 AI 方向值不值得做"的前置文档。
· 市场容量、竞品矩阵、差异化卖点
· ROI 估算、风险评估、Go/No-Go 决策

→ GitHub fork

TEMPLATE · 02
MRD 模板 · Market Requirement

用户研究 + 需求收敛产物，BRD 通过后做。
· 用户分层、JTBD 拆解、需求池打分
· 用户旅程、痛点矩阵、需求优先级

→ GitHub fork

TEMPLATE · 03
PRD 模板 · Product Requirement

给工程师可执行的产品定义，AI 版本含 harness 章节。
· Agent 边界、工具列表、评测指标
· 上下文构造、人机交接、降级方案

→ GitHub fork

TEMPLATE · 04
Design Spec · 设计规范

交付给设计的视觉与交互定义，AI 产品专项。

· 流式输出、可控/可停、引用与可信度
