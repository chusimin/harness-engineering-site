§1 · 被低估的真相

同一个模型，
为什么产品差这么远？

2025 年起，AI 产品的胜负手悄悄变了。 过去问的是"谁的模型更强"，现在三个截然不同的产品，背后可能是同一个模型。

§2 · 一句话定义

Harness = 那一整圈代码。

一个 agent 从来不是"一个很强的模型"。它是 —— 一个模型，加上一套把模型驾驭起来的代码。

它不是什么 —— ❶ 不是模型本身（模型是引擎，harness 是底盘）。 ❷ 不只是提示词（提示词只是 harness 最里面的一层）。 ❸ 不是某个框架（LangGraph、Claude Agent SDK 这些是"用来搭 harness 的工具"，不等于 harness 本身）。

§5 · 尺度

它有多大？
三个尺度，套着的。

这是板块① 最重要的一个框架。 Harness 不是固定大小的东西 —— 它有三个尺度，一个套一个。 各家厂商定义看起来打架，其实只是站在这把"尺度尺"的不同刻度上。

Harness 不是越大越好，是"刚好够用"。 模型一升级，本来需要任务级编排的活，可能塌回组件级一个 agent 就能干。 —— Anthropic 在官方文章里点出的关键。
§6 · 边界

跟提示词工程、
上下文工程是什么关系？

这三个词经常被混用。它们的关系不是并列 —— 是层层嵌套。

§7 · 一句话总结
Harness 是包在模型外面的那一整圈代码与结构， 它把被动的模型变成能干活的 agent； 它的构成可记成「转 · 知 · 做 · 检」； 它有「组件 / 任务 / 组织」三个尺度； 各家定义看似打架，其实只是站在这把"尺度尺"的不同刻度上。

02
它从哪来
从 prompt engineering 到 harness engineering —— 一条 4 年的概念演化谱系
4 个可视化 约 12 分钟
§1 · 开篇

这个词，
是凭空冒出来的吗？

2026 年初，"Harness Engineering" 几乎一夜之间出现在所有人嘴边：OpenAI 发文、Anthropic 发文，连软件工程界的圣地 Martin Fowler 的网站都在谈它。

很自然会有两种反应：一种觉得"这是真突破"，一种觉得"AI 圈又在炒新词"。 要判断它是哪一种，得先回答一个问题：这个词，到底从哪来的？

答案分两层 —— harness 这个词很老、很朴素；但 "Harness Engineering" 作为一个热词，非常新，新到我们能一帧一帧倒回去看它怎么火起来的。
§2 · 第一层

harness 是个
朴素的老词。

harness 这个词，在 AI 之前就在软件工程里待了很久 —— 而且一直很低调。 它的老家是软件测试。

§3 · 第二层

"Harness Engineering" 的
七周。

harness 是老词。真正新的，是有人把 "Harness" 和 "Engineering" 焊在一起，让它变成一个全行业的词。这件事就发生在最近，而且能精确复盘 —— 一个大约七周的故事，分五幕。

§4 · 微妙的真相

用到的技术，
竟然没有一个是新的。

如果你把这段历史复盘完，会发现一件很微妙、也很关键的事： Harness Engineering 用到的技术，竟然没有一个是新的。 Linter 代码检查、把大任务拆成小步、给产出做质量评估 —— 这些做法早就有了，很多工程师本来就天天在做。

工程领域真正的进步，常常不在于发明了什么新技术，而在于有没有一套统一的框架，把散落的能力组织成"可以系统设计、可以持续优化"的方法。
§5 · 收获

这段谱系，
给你两件武器。

知道"它从哪来"，不只是讲故事 —— 它给你两样实用的东西：

harness 是个朴素的老词，内核从未变过 ——"为支撑某件事而搭的框架"。 "Harness Engineering" 是个崭新的热词，从个人博客到行业共识只用了约七周。 它没发明任何新技术 —— 它的价值在于把散落的老做法组织成了一套可系统设计的方法。
