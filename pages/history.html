04 · LINEAGE

谱系

从 prompt 到 harness 的 4 年命名演化史 —— 用一个具体案例「修登录页 500 错误」串联三层，看为什么前一层不够用。

01 · 4-YEAR EVOLUTION

4 年演化

每一层都比上一层管得更多。从「怎么问得好」→「给什么上下文」→「怎么包一整套能跑能管的系统」。

2022 · PROMPT ENGINEERING
提示词工程起点
GPT-3 时代。研究怎么写好一条 prompt 让模型回答得更好 —— few-shot、CoT、角色扮演。 Brown et al. 2020 (GPT-3)
2023 · TOOL USE
ReAct + Function calling 引入
模型开始"动手" —— 推理-行动循环、函数调用。仍是 prompt-centric，但已经有了 agent 的雏形。 Yao et al. (ReAct)
2024 · CONTEXT ENGINEERING
"上下文工程"命名出现
RAG、长上下文、记忆系统成熟。社区开始把"装桌面"这件事看成独立学问。 LangChain · Context Engineering
2025–2026 · HARNESS ENGINEERING
Harness 工程正式定义
Anthropic 与 OpenAI 在同一时间窗口先后撰文，正式把"包在模型外的一整圈代码"立成工程领域。 Anthropic 2026 OpenAI 2026

同一个案例「修登录页 500 错误」 —— 下面分别用三层的能力去试着干，看每一层在哪儿掉链子、下一层为什么必须出现。

2022—2023 · LAYER 1

提示词工程 · 能问，但只能问

这一层只研究"怎么写好一条 prompt 让模型回答得更好"。

提示词工程是大模型时代第一个被命名的工程实践。它把"和模型对话"当成一门可以打磨的手艺，研究内容主要包括：

  • Few-shot examples —— 在 prompt 里给几个示例，让模型照葫芦画瓢。
  • Chain-of-thought —— 让模型"分步想"，提高复杂推理的准确率。Wei et al. 2022
  • 角色扮演 prompt —— "你是一个资深工程师 / 律师 / 心理咨询师…"
  • ReAct 模式 —— 让模型边推理边输出动作，开启了工具调用的雏形。Yao et al. 2023

这一时期的研究主线是"输入 → 输出"这单次调用怎么调得更好。模型还没有持续状态、不能调用工具、不能改外部世界。

✓ 能干什么
  • · 让模型回答问题
  • · 解释概念
  • · 生成草稿
  • · 单次推理任务
✗ 不能干什么
  • · 读你的代码库
  • · 跑你的测试
  • · 改你的文件
  • · 跨调用记得上一轮

↓ 用提示词工程去试着干「修登录页 500 错误」——

提示词工程 · 修登录页 500 错误
⚠ 为什么这一层不够
提示词工程让模型变成了一个聪明的"答题机器"，但它没有手、没有眼睛、没有记忆。修一个真实 bug 需要的不只是聪明的回答 —— 它需要去读代码、跑日志、试改、验证。这正是下一层「上下文工程」要解决的事。
2024 · LAYER 2

上下文工程 · 能看见，但还是只能建议

这一层研究"给模型什么上下文" —— 从 RAG 检索到记忆系统，让模型"看见"它本来不知道的东西。

"Context Engineering" 这个词在 2024 年正式被社区命名。它的研究内容比提示词工程外扩了一圈：

  • RAG 检索 —— 把外部知识库切片、向量化，运行时检索相关片段拼进上下文。
  • 文档分块策略 —— chunk size、重叠、metadata 过滤…
  • 对话记忆 —— 短期 / 长期记忆、摘要压缩、用户偏好持久化。
  • 工具结果反馈 —— Function calling 拿到结果后，怎么塞回 context、怎么让模型继续推理。OpenAI Function Calling

换句话说，这一层的关注点从"怎么问"变成了"把什么塞到模型桌面上"。模型有了"眼睛"，但还是只能看见 + 输出建议。 LangChain · The Rise of Context Engineering

✓ 能干什么
  • · 根据真实代码库给答案
  • · 引用历史对话和文档
  • · 调用工具获取数据
  • · 多轮对话保持连贯
✗ 不能干什么
  • · 真的去执行修改
  • · 跑测试验证
  • · 回滚错误的尝试
  • · 自主完成多步任务

↓ 用上下文工程再试一次「修登录页 500 错误」——

上下文工程 · 修登录页 500 错误
⚠ 为什么这一层还不够
上下文工程让模型"能看见"你的代码，但还是只能给建议。修一个真实 bug 需要它真的去动手、试错、验证、回滚。光靠"装桌面"没有"循环 + 边界 + 评测"，模型永远停在"我建议你…"。这正是下一层「驾驭工程」要解决的事。
2025—2026 · LAYER 3

驾驭工程 · 能真的把活干完

这一层研究"怎么包出一个能跑、能管、能评的 Agent 产品" —— 完整的 agent 循环 + 工具执行 + 沙箱 + 评测 + 多 agent 编排。

2026 年初，Anthropic 与 OpenAI 在同一时间窗口先后撰文，正式把这个工程领域立名。它在上下文工程外面再包了一圈，把所有"让 agent 真的能跑起来"的东西都拢进来：

  • Agent 主循环 —— 模型 → 工具调用 → 结果回流 → 再来一圈，直到任务完成或触发停机。
  • 工具执行 + 沙箱 —— 真的去改文件、跑命令，但在隔离环境里、有审批 hook。
  • 评测 + 可观测 —— Trace、token 计账、回放、AB 测试、监控告警。
  • 多 agent 编排 —— 子任务调度、专家路由、协作记忆。
  • 长周期治理 —— 跨周/跨月任务、记忆分层、组织级策略。Anthropic · Harness Design OpenAI · Harness Engineering

到了这一层，关注点从"怎么让一次推理更好"变成了"怎么让一支 agent 队伍长期、安全、可观测地干活"。

✓ 能干什么
  • · 自主完成多步任务
  • · 调用工具、执行命令
  • · 自我验证、试错回滚
  • · 全程可观测、可回放
→ 下一个边界
  • · 多组织 agent 协作
  • · 跨产品长期治理
  • · 可证明安全

驾驭工程 · 修登录页 500 错误
✓ 任务真的修完了
从"看见"到"动手" —— 这才是 harness 工程的能力。同一个 bug，前两层只能给建议，到了这一层，模型自主完成了 read → diff → edit → test → lint 全流程，并留下了可回放的 trace。这就是 Agent = Model + Harness 的真正含义。

06 · WHAT'S NEXT

下一个名字会是什么

下一个名字会是什么？没人知道。但如果历史规律成立，它一定是因为 harness 工程的"控制面积"不够了 —— 而要管的东西又多了一圈。

推测，非定论。 下面三个名字都是社区讨论中出现过的候选概念，没有任何一个是"已被锁定的下一层"。把它们列出来，是为了帮你提前感受 harness 工程的下一个边界在哪。

AGENT SOCIETY
ENGINEERING
智能体社会工程
多组织、多 agent 网络的治理 —— 当一家公司的 agent 要和另一家公司的 agent 协商、谈判、签合同，单一组织的 harness 就不够用了。
AI SYSTEM
ENGINEERING
AI 系统工程
模型 + harness + 产品的统一系统观 —— 把模型训练、harness 编排、产品上线、用户运营当成一个不可分割的系统去设计。
TRUSTLESS AGENT
ENGINEERING
无信任智能体工程
加密验证、可证明安全的 agent —— 在不信任前提下让多 agent 协作的工程。可能借鉴零知识证明、TEE、形式化验证等思路。

历史告诉我们：每一层新名字的出现，不是因为发明了新东西，是因为旧名字管不住了。Prompt Engineering 管不住"装桌面"，于是有了 Context Engineering；Context Engineering 管不住"循环 + 边界 + 评测"，于是有了 Harness Engineering。

下一个名字也一样 —— 它会在当前的 harness 工程管不住某件事的那一刻自然涌现。你只需要保持观察：什么时候开始有人抱怨"harness 工程也不够用了"。
