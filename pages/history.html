06 · GENEALOGY

谱系

从 prompt 到 harness 的 4 年命名演化史 —— 每一次重命名,都是上一层"管不动了"的信号。

01 · TIMELINE

4 年,4 次改名

每一次"重命名",都是因为上一层名字罩不住新出现的工程问题了。

ERA 01 · 2022
2022

Prompt Engineering · 起点

GPT-3 出圈,人们发现同一个模型,换一种问法,答案差天差地。"prompt engineering" 一词在社区流行起来 —— 重点是怎么措辞:few-shot 示例、chain-of-thought、role play。这一阶段的"工程",几乎全部发生在一个文本框里。
代表文献:GPT-3 paper (Brown et al., 2020)、社区指南 Prompt Engineering Guide。
ERA 02 · 2023
2023

Tool Use / Function Calling 引入

ReAct 论文把"推理-行动"循环写清楚,OpenAI 推出 function calling API。模型第一次能"输出 → 调一个真实函数 → 拿结果接着想"。"prompt" 一词开始不够用 —— 现在还要管工具集、工具描述、错误重试,这些都不是"措辞"问题了。
代表文献:ReAct (Yao et al., 2022)、OpenAI Function Calling (2023-06)。
ERA 03 · 2024
2024

Context Engineering 命名出现

长上下文(100k+)、RAG 普及、记忆系统兴起。Andrej Karpathy、Anthropic、Cohere 等人在 2024 反复用 "context engineering" 来概括:与其调 prompt,不如重新设计"给模型看什么"。这一阶段把RAG、记忆、工具描述、系统提示都包了进来。
代表文献:Anthropic · Contextual Retrieval (2024-09)、Cohere RAG docs。
ERA 04 · 2025–2026
2025 — 2026

Harness Engineering 正式定义

Anthropic、OpenAI 同一季度,独立提出 "harness" 这个词。上下文工程罩不住"8 小时长任务、多 agent 协作、组织级运行"这种新问题 —— 需要一个更外层的概念。harness = 主循环 + 上下文 + 工具 + 评测 + 沙箱 + 可观测,一整圈包出来,把模型变成可运行、可治理、可迭代的 Agent 产品。
代表文献:OpenAI · Harness Engineering (2026-02-11)、Anthropic · Harness Design (2026-03-24)。
02 · RENAMING LOGIC

每一次重命名,背后的逻辑

不是术语炒作。每次都是因为上一个名字罩不住一类新出现的工程问题。

Prompt → Tool Use
从"措辞"到"动手"

触发:模型只能输出文字,做不了实事。 新增问题:工具注册、参数 schema、错误重试、安全围栏。 关键词从 "怎么说" 变成 "怎么做"。

里程碑:ReAct 论文 + OpenAI function calling。

Tool Use → Context Eng.
从"单次调用"到"整套上下文"

触发:工具一多,上下文塞不下;模型读不准了。 新增问题:RAG 检索、记忆压缩、上下文窗口预算、相关性排序。 关键词从 "调哪个工具" 变成 "塞什么进去"。

里程碑:Anthropic contextual retrieval。

Context Eng. → Harness Eng.
从"一次对话"到"长任务 + 治理"

触发:8 小时长任务、多 agent、生产环境治理来了。 新增问题:状态机、retry/rollback、评测集、沙箱、成本治理、可观测。 关键词从 "怎么答好" 变成 "怎么跑得稳"。

里程碑:OpenAI 2026 + Anthropic 2026。

注意:这不是"取代",而是"包裹"。harness 工程师该会的 prompt 技巧、context 设计,一个都不能少;它只是把这些都放进了更大的工程上下文。这一点很重要 ——下一节专门讲。

03 · NESTED LAYERS

三层嵌套

这不是术语革命,而是同心圆扩张。新名字把旧名字包在里面,而不是取代。

最内层
提示词工程 · Prompt Engineering
研究 "怎么写好一条 prompt"
⊂
中间层
上下文工程 · Context Engineering
研究 "给模型什么上下文"(含 RAG、记忆、工具)
⊂
最外层
Harness 工程 · Harness Engineering
研究 "包出一个能跑、能管、能评的 Agent 产品"

每一次外扩,都是因为前一层"控制面积"不够了。外层包内层,谁也没消失。一个合格的 harness 工程师,必然是合格的 context 工程师、prompt 工程师 —— 反过来不成立。

04 · WHAT'S NEXT

下一个名字会是什么?

顺着同一套逻辑外推 —— 当 harness 也罩不住的那一天,新词会指向什么?

"harness 工程" 罩住了 一个产品 + 一支 agent 队伍 的工程边界。但当 agent 跨产品、跨组织、彼此协议化对话(MCP 已经在做),问题就会再次溢出:不再是"我家这一支队伍跑得好不好",而是"agent 与 agent 之间的协议、信任、市场怎么定"。

合理的下一个名字,可能是 Agent Society Engineering(智能体社会工程)或者 Agent Economy Engineering(智能体经济工程)。研究对象会从"一个 harness"扩到"多个 harness 之间的协议、市场、声誉、争议解决"。MCP、AGENTS.md 已经是这一层的雏形。

如果这条线是对的,那今天投资在 harness 上的工程范式,会自然过渡进去 —— 就像 prompt 技巧没在 context 时代消失,context 技巧也不会在 harness 时代消失。名字一直在向外扩,工具栈一直在向下兼容。

SPECULATION
推测,非定论。"Agent Society Engineering" 这个词目前未被头部实验室正式命名,这里只是顺着 prompt → context → harness 的扩张轨迹做的外推。读者请把它当作思考方向,而非已成事实。
